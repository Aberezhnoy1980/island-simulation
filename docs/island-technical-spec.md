# Техническое задание: симуляция острова

## 1. Цель проекта

Реализовать консольное (на первом этапе) приложение-симуляцию острова на Java, демонстрирующее уверенное владение ООП и ключевыми темами «Java Core».  
Проект одновременно является учебным (закрепление пройденного материала) и витринным (production-like репозиторий для портфолио на GitHub).

## 2. Общая концепция

Модель представляет собой «остров» — двумерное поле локаций (клеток), например 100×20.  
Каждая локация может содержать:

- растительность (ресурс, которым питаются травоядные);
- набор животных различных типов (хищники, травоядные, возможны дополнительные типы).

Во времени происходит симуляция «жизненного цикла» острова, шагами:

- животные перемещаются между локациями;
- животные едят (растительность или других животных);
- животные размножаются;
- животные умирают от голода, возраста или других условий.

Параметры мира и существ должны быть конфигурируемыми (количество видов, размеры популяций, скорость размножения, лимиты на локацию и т. д.).

## 3. Стек и технические ограничения

- Язык: Java 21 (без использования Spring и других DI/веб-фреймворков).
- Сборка: Maven (один модуль на первых этапах, возможна дальнейшая модульность при необходимости).
- Запуск: консольное приложение (JAR), точка входа `main`.
- Разрешено и поощряется:
  - использование продвинутых коллекций и стандартных структур данных;
  - использование Stream API;
  - использование современных возможностей Java 21 (records, sealed classes, pattern matching и др., если они уместны);
  - использование многопоточности (как минимум `ExecutorService`, желательно ForkJoinPool, отдельные потоки для логики симуляции/логирования).
- Нельзя:
  - использовать Spring и аналогичные фреймворки;
  - тянуть тяжелые внешние библиотеки без аргументированной пользы (баланс: учебный проект, но production-like).

## 4. Основные требования к архитектуре и ООП

Цель — максимально продемонстрировать грамотное ООП и гибкую архитектуру.

### 4.1. Абстракции и расширяемость

- Обязательна четкая иерархия сущностей:
  - базовые интерфейсы/абстрактные классы для «животного», «растительности», «сущности острова»;
  - отдельные реализации для конкретных видов животных.
- Добавление нового типа животного или другого участника симуляции должно требовать **минимальные изменения** в существующем коде:
  - по возможности — только новая реализация/класс + конфигурация;
  - минимум изменений в управляющем коде.
- Использовать композицию, полиморфизм, инкапсуляцию, SOLID-принципы там, где это уместно.

### 4.2. Создание и конфигурирование объектов

- Использовать паттерны:
  - фабрика/абстрактная фабрика (для создания животных, локаций, мира в целом);
  - билдеры для сложных конфигураций (например, конфигурация острова, сценариев).
- Конфигурация должна быть отделена от бизнес-логики:
  - настройки мира и видов читаются из конфигурационных файлов (YAML/JSON/Properties — формат можно уточнить);
  - предусмотрена возможность подменять конфигурацию без изменения кода.
- Желательно:
  - использование аннотаций (например для считывания конфигураций для классов при создании)
  
### 4.3. Организация доменной модели

- Четкое разделение:
  - доменная модель (Island, Location, Animal, Plant и т. п.);
  - сервисы/компоненты симуляции (движок шага, сервис питания, сервис перемещения, сервис размножения);
  - инфраструктурный слой (логирование, конфигурация, persistence — по мере необходимости).
- Избегать «божественных» классов, где сосредоточена вся логика.

## 5. Логика симуляции

### 5.1. Сущности

Минимальный набор:

- Остров (`Island`) — агрегирует локации.
- Локация (`Location`) — содержит коллекции животных и растительности.
- Базовый тип животного (`Animal`) — общее поведение и свойства:
  - вес, скорость, максимальная насыщаемость, шанс размножения, и т. д.;
- Конкретные виды животных (набор видов определяется конфигурацией и/или заданием).
- Растительность (`Plant`) — отдельная сущность (может иметь скорость роста, максимальную плотность на клетку и т. п.).

### 5.2. Действия на шаг симуляции

Для каждого «тика» симуляции:

- Передвижение животных между локациями (учитывая скорость и ограничения).
- Питание:
  - животные едят растения;
  - хищники едят травоядных (с вероятностями, ограничениями по весу и т. д.).
- Размножение:
  - при выполнении условий (сытость, наличие пары, не превышен лимит на локацию).
- Смерть:
  - от голода (отсутствие еды несколько шагов);
  - от естественных причин (опционально: возраст);
  - от съедения хищником.

Логика должна быть организована так, чтобы можно было конфигурировать поведение через параметры и не захламлять код условиями `if/else` по типам.

## 6. Многопоточность

- Проект должен демонстрировать осознанное применение многопоточности:
  - возможен вариант: обработка отдельных локаций острова параллельно (например, через ForkJoinPool);
  - либо разделение потоков по задачам (движок симуляции, логирование/статистика, сохранение состояния).
- Нужно аккуратное обращение с общей mutable-состоянием (коллекции локаций, животных):
  - Thread-safe подходы, использование синхронизации или конкурентных коллекций при необходимости;
  - избежание гонок и неконсистентных состояний.
- Многопоточность не должна ломать читаемость архитектуры, но должна быть видимой и оправданной.

## 7. Конфигурация и управление параметрами

- Конфигурационные файлы:
  - отдельный файл/файлы для:
    - общих параметров симуляции (размер острова, количество шагов, задержки между шагами, уровень логирования и т. д.);
    - параметров видов (масса, скорость, диета, вероятность успеха при охоте, максимальное количество на локации);
  - формат: YAML/JSON/Properties (выбрать и зафиксировать в ТЗ).

## 8. Качество кода и инфраструктура

### 8.1. Структура репозитория

- Maven-проект, стандартная структура `src/main/java`, `src/test/java`.
- Папка `docs/` с:
  - данным ТЗ;
  - доп. документацией (описание архитектуры, диаграммы по мере необходимости).

### 8.2. Тестирование

- Приоритет тестов: средний+.
- Минимум:
  - модульные тесты для ключевых доменных классов и сервисов (логика питания, размножения, передвижения, конфигурации);
  - smoke-тесты запуска симуляции на нескольких шагах.
- Технологии:
  - JUnit (версия по умолчанию для Java 21 и Maven);
  - по необходимости — Mockito или аналог для моков (если не противоречит духу «минимального» стека).

### 8.3. CI/CD

- Настроить базовый CI на GitHub Actions:
  - сборка Maven;
  - прогоны тестов при push/PR в основную ветку;
  - опционально — сборка артефакта (JAR).
- Логика пайплайна должна быть простой и прозрачной, без избыточных этапов.

### 8.4. Код-стайл и документация

- Соблюдать единый код-стайл (по возможности — включить линтер/форматтер через Maven-плагин).
- Использовать JavaDoc для ключевых публичных классов и методов.
- Написать понятный `README`:
  - краткое описание проекта;
  - как собрать и запустить;
  - краткое описание архитектуры и основных фич;
  - ссылка на ТЗ и листиниг команд.

## 9. UI и варианты развития

На начальном этапе UI — консольный:

- запуск симуляции из `main`;
- вывод ключевых метрик по шагам (статистика популяций, события в симуляции);
- опционально — простое меню/параметры запуска.

После реализации чистой архитектуры и стабильной логики возможны варианты развития:

- JavaFX-приложение с визуализацией острова;
- Web-UI (отдельное приложение/служба поверх текущего ядра, в том числе. например, с шаблонизаторами);
- продвинутая консоль (цвета, псевдографика, интерактивные команды).
- **Псевдографика и Unicode:** при визуализации клеток на консоли допустимо отображать виды эмодзи/символами Unicode (как в формулировке задания: 🐃, 🐻, 🐎, 🦌, 🐗, 🐑, 🐐, 🐺, 🐍, 🦊, 🦅, 🐇, 🦆, 🐁, 🐛 и т. п.). Имеет смысл вынести таблицу «идентификатор вида из конфига → символ» в отдельный ресурс или класс (`SpeciesGlyph`, `application.yml` и т. д.), чтобы не размазывать литералы по коду. Реализацию можно отложить до появления вывода карты; на архитектуру ядра симуляции это не влияет.
- использование профилей для разных режимов работы (например, dev/demo/perf);
- возможность частично менять параметры в рантайме (если есть продвинутый UI - переключать режим логирования, количество шагов симуляции, скорость, размеры видов и т. п., если это не усложняет архитектуру).
- сериализация состояния (снапшоты симуляции в файл) — для последующего анализа;
- возможность запускать симуляцию с разными профилями (например, короткая демо-симуляция, длительная нагрузочная, etc.).

Выбор направления UI будет обсуждаться отдельно после готовности базовой симуляции.

## 10. Опциональные источники данных и БД

Базовая (обязательная для MVP) схема: параметры симуляции и каталог видов читаются из **файлов** (YAML/JSON/Properties) на classpath или с диска — см. раздел 7.

**Расширение (необязательное):** вынести часть данных во внешнее хранилище, например:

- каталог видов и численности на старте;
- сохранённые снапшоты состояния острова для анализа или продолжения прогона;
- журнал агрегированной статистики по шагам.

### 10.1. Сложность без Spring и без «тяжёлого» ORM

- **Spring Data / JPA «в лоб»** без Spring-контекста на практике не используют; полноценный Hibernate/JPA вручную (Persistence API, EntityManagerFactory) возможен, но резко увеличивает объём инфраструктурного кода и отвлекает от предметной области симуляции.
- **Разумный компромисс для учебного проекта:** если БД нужна, предпочтительны **JDBC** (или очень тонкая обёртка) + явный слой репозиториев/DAO: `DataSource`, try-with-resources, маппинг строк результата в DTO/records. При необходимости — пул соединений (например HikariCP) без контейнера.
- **Миграции схемы** (Flyway/Liquibase) — по желанию, если появится реальная схема таблиц; для прототипа допустимы скрипты в `resources/db` и ручной порядок применения.

### 10.2. Выбор источника при старте

Чтобы не смешивать file-based и JDBC в одном неделимом классе, в ТЗ закрепляется идея **абстракции источника конфигурации/данных**, например:

- интерфейс в духе `SimulationConfigLoader` / `SpeciesCatalogProvider` с реализациями `ClasspathYaml*`, `FileYaml*`, опционально `Jdbc*`;
- выбор реализации: аргументы командной строки, переменные окружения и/или корневая секция главного конфига (`config.source: file` | `jdbc`, параметры подключения).

**Целесообразность:** вводить БД **после** стабилизации доменной модели и сценариев с файловой конфигурацией, чтобы границы домена и инфраструктуры были уже ясны.

## 11. Контейнеризация (Docker, Compose)

- Для варианта «только JAR + файлы конфигурации» Docker **не обязателен**, но полезен для воспроизводимой среды (`mvn package` в образе, одинаковый JDK) и как привычная практика для портфолио.
- **Если подключается СУБД:** связка **Docker Compose** (приложение + PostgreSQL или аналог) — рекомендуемый целевой вариант: изоляция версии БД, один `docker compose up` для демонстрации.
- **Отдельный контейнер/сервис под метрики или централизованные логи** (Prometheus, Loki и т. п.) — опциональное развитие, не входит в MVP; имеет смысл только при появлении реальных требований к наблюдаемости, а не «ради галочки».

Инфраструктурные артефакты (`Dockerfile`, `compose.yaml`) описываются в репозитории отдельно от доменного кода; приложение по возможности остаётся запускаемым и без контейнеров.
